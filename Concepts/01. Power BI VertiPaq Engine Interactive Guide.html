Introduction to VertiPaq Engine

Welcome to this interactive training on Power BI's VertiPaq Engine - the powerhouse behind Power BI's blazing-fast performance. In this guide, we'll explore how VertiPaq stores and processes data with incredible efficiency.

The Two Engines of Power BI

Power BI uses two complementary engines when processing your data:

Formula Engine

The "Brain" - interprets your DAX formulas, creates execution plans, and manages the overall query process.

Characteristics:

  • Single-threaded (works on one task at a time)
  • Interprets DAX queries
  • Creates execution plans
  • Requests data from storage engine
VertiPaq Engine

The "Muscle" - stores compressed data in memory and performs high-speed scans when requested by the Formula Engine.

Characteristics:

  • Multi-threaded (parallel processing)
  • Columnar storage format
  • Advanced compression
  • In-memory operation

Analogy: Think of the Formula Engine as a project manager who plans the work, and VertiPaq as the specialized team that executes the actual tasks at lightning speed.

The VertiPaq engine is only activated when your data is in Import Mode. If you're using DirectQuery, a different storage engine is used. This is why Import Mode is typically much faster for complex calculations.

Columnar Storage: The Core Concept

Traditional databases store data row-by-row, like reading an entire book to find a specific fact. VertiPaq uses columnar storage, which organizes data by columns instead of rows.

Bookstore vs. Reference Library Analogy
Row-Based Storage

Like a Bookstore where books (rows) are stored together. To find all authors (a column), you must open every book and skip to the author page.

Best for:

  • Transaction processing
  • Updating individual records
  • Reading complete rows

Not ideal for:

  • Aggregate calculations
  • Scanning specific columns
  • Analytical queries
Columnar Storage

Like a Reference Library where all author pages are filed together, all title pages together, etc. To find all authors, you go directly to the "author" shelf.

Best for:

  • Aggregate calculations
  • Scanning specific columns
  • Analytical queries
  • Data compression

Not ideal for:

  • Transaction processing
  • Updating individual records

Columnar storage: Each column is stored separately and contiguously

This columnar approach gives VertiPaq two major advantages:

  1. Faster Aggregation: When calculating a sum of sales, VertiPaq only needs to scan the "Sales Amount" column.
  2. Better Compression: Similar values in a column can be compressed more efficiently (like run-length encoding).

Compression Techniques

VertiPaq uses multiple compression techniques to store data efficiently in memory. The engine automatically chooses the best method for each column based on its data characteristics.

Dictionary Encoding

Most Common

For: Text columns, high-cardinality numbers

Analogy: Like a numbered menu in a restaurant. Instead of writing "Grilled Chicken Sandwich" every time, the kitchen just uses order #42.

Example: "New York", "London", "Tokyo" become 0, 1, 2

Every unique value gets an ID, and only the IDs are stored in the column.

Value Encoding

For Numbers

For: Integer, currency, decimal columns

Analogy: Like measuring from a baseline. If chairs are 100-110cm tall, store them as 0, 1, 2, 3... representing cm above 100.

Example: Values 1001, 1002, 1003 become 1, 2, 3 (with base 1000)

Finds the smallest value, stores others as differences from that baseline.

Run-Length Encoding (RLE)

For Repeats

For: Columns with many repeating consecutive values

Analogy: Like a tournament bracket showing "Team A advances to round 3" instead of listing Team A 3 times.

Example: "A, A, A, B, B, C" becomes "A[3], B[2], C[1]"

Stores the value once with a count of how many times it repeats consecutively.

Dictionary Encoding Deep Dive

Dictionary Encoding is VertiPaq's most important compression technique. It's how text and high-cardinality numeric values are efficiently stored.

Key Insight: Instead of storing "Jake" (4 characters = ~4 bytes) multiple times, we store the number 0 (which might use just 1-2 bits with bit-packing).

The Cardinality Factor

Cardinality refers to the number of unique values in a column. This is the most important factor affecting VertiPaq's performance and compression.

Practical Implication: When designing your data model, be mindful of high-cardinality columns. Sometimes you can reduce cardinality by:

  • Removing unnecessary precision (rounding decimals)
  • Grouping values into categories
  • Using integers instead of GUIDs where possible
  • Only including columns you actually need for analysis

Relationship

A "cross-reference" connecting ProductID in Sales table to ProductID in Product table. This allows VertiPaq to quickly look up product details without storing them in both tables.

Key Components Explained

Segments

Data is divided into segments of ~8-16 million rows each. Segments are the unit of scanning during queries - VertiPaq can scan multiple segments in parallel.

Partitions

A group of segments, typically used to divide data by date ranges (e.g., by year). Partitions can be processed and refreshed independently.

Hierarchies

Pre-defined drill-down paths (like Year → Quarter → Month → Day) that VertiPaq can use to answer queries faster without recalculating relationships.

Relationships

Connections between tables that tell VertiPaq how to join data. These are stored as special data structures for rapid lookup during queries.

Important Note: Calculated columns are stored just like regular columns - they consume memory and go through the same compression process. This is different from measures, which are calculated at query time.

Optimization Tips & Best Practices

Understanding VertiPaq's internals helps you design better data models. Here are practical recommendations based on how VertiPaq works:

Do This

  • Use Import Mode to leverage VertiPaq's speed
  • Remove unnecessary columns - each column adds overhead
  • Use integers instead of text for IDs/codes
  • Group high-cardinality values into categories
  • Use star schema for optimal relationship performance

Avoid This

  • Including unique IDs unless absolutely necessary
  • Using text for flags (use 1/0 instead of "Yes"/"No")
  • Too many calculated columns - they consume memory
  • Wide tables with many columns (aim for narrow, long tables)
  • Unnecessary precision in decimals (round when possible)
Data Type Recommendations
Data Type VertiPaq Efficiency When to Use Alternative to Consider
Integer (Whole Number) Excellent Counts, IDs, categories, flags —
Decimal/Fixed Good Currency, precise measurements Consider rounding to 2 decimals
Text (Low Cardinality) Good Categories with few unique values Use integer codes if possible
Text (High Cardinality) Poor Names, descriptions, comments Only include if essential for analysis
Date/Time Excellent Timestamps, dates Split into separate date/time columns if needed
Boolean Excellent True/False flags Use 1/0 integers for even better compression

Remember: The goal is to help VertiPaq compress your data as efficiently as possible. Smaller data size means more data fits in memory, and scanning happens faster. Use tools like DAX Studio's VertiPaq Analyzer to see exactly how your columns are being compressed and where you can optimize.
